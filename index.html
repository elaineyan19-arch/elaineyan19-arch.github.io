AI in Practice

I don't wait for the tools to mature.

I build with what exists now.

The thing I've learned that most people haven't said out loud yet

"I use AI to move fast. I use my domain expertise to get it right. The model gets me 70% of the way. The last 30% is still mine."

I learned this the hard way in 2023. I used ChatGPT and DeepSeek to enter e-commerce insurance from zero — I had no background in it at all. The AI compressed weeks of research into days. Competitive landscape, regulatory structure, business model mechanics — I had working knowledge fast enough to actually be useful. But the model was also confidently wrong in ways that only an expert would catch. I caught them because I kept domain experts in the loop at every stage.

We shipped. The product generated millions in GWP within months.

That experience set the rule I've worked by since: AI changes the speed. It doesn't change who's responsible for getting it right.

Personal · 2026

Production job search agent — Claude API + GitHub Actions + Gmail SMTP

Python · Anthropic SDK · GitHub Actions · Gmail SMTP · JSON dedup state

Built and deployed a fully automated daily job agent from scratch. It calls the Claude API with a structured search prompt, deduplicates results against a persisted JSON store so no role appears twice, applies role-matching logic, and delivers a formatted HTML digest via Gmail SMTP on a GitHub Actions cron schedule. Zero manual trigger. Runs in production every weekday morning. Cost: ~$0.007/run.

This is what building with AI actually looks like — not a demo, not a prototype. Prompt engineering, state management, error handling, production deployment. The same process I'd apply to any 0→1 product.

Learning · 2025–2026

Learning to think like a builder

I'm working through Andrej Karpathy's technical content and Andrew Ng's courses to build a foundation — not credentials, but a way of thinking. Understanding where models fail. Why they hallucinate. What that means for the product decisions you have to make on top of them.

I also spend time on GitHub reading how practitioners structure their skills files and prompt context — not to copy, but to take inspiration and build my own tailored versions for how I actually work.

The part I find most interesting is thinking through user scenarios from first principles: where does AI genuinely change what's possible for someone? Where does it create new friction? Those questions are harder than the tooling, and they're the ones I'm trying to get better at.

I'm not there yet. But that's the work.
